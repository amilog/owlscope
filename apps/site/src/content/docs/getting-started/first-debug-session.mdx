The fastest way to learn a debugger is to watch one solve a problem. This page
walks through a real session — a bug we hit while building OwlScope itself.

## The bug

Users reported that on Android, network responses sometimes appeared empty in
the desktop UI even though the app clearly received them. No errors, no
warnings — just blank.

## Reproducing

Open OwlScope, run the app on a physical Android device, hit the failing
endpoint. A `network:response` event arrives with status 200 but the
**Response** tab is empty.

## Step 1 — Errors panel

Empty. The bug doesn't throw. It just silently drops the body.

## Step 2 — Network panel

The request is there. Status, headers, timing all look right. Switching to
Response: blank.

## Step 3 — Inline detail

Clicking the row expands the inline detail. The **Headers** tab shows
`Content-Type: application/json; charset=utf-8`. So far so good.

## Step 4 — Logs panel

Filter to `level=error`. Nothing related. But scrolling earlier, we spot a
`console.warn` from the SDK: `tolerantPrettifyJson skipped — body > 64KB`.

That's the smoking gun. The Flutter SDK was capping bodies at 64KB. The
endpoint returned ~120KB. The truncated bytes weren't valid JSON, so the
desktop's `JSON.parse` failed silently.

## The fix

Bumped the SDK's default body limit from 64KB to 16MB, and made it
configurable via `httpMaxBodyBytes`. Added a tolerant JSON pretty-printer for
truncated payloads so the user at least sees indented text instead of an
escape-laden one-liner.

## What this shows

OwlScope didn't tell us the answer — it gave us the **observability** to find
it ourselves. Five panels, three minutes, root cause identified.

That's the loop:

1. **Errors** — anything thrown?
2. **Network** — anything failing?
3. **State** — anything weird in the store?
4. **Logs** — anything unusual?
5. **Performance** — anything slow?

When all five are quiet, you've shipped well.
